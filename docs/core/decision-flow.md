# Decision FlowThis document defines the **end-to-end lifecycle** of a decision in Manthan.It describes how data flows from external input into the deterministic core and produces a final outcome.---# 🧠 High-Level Flow```txt(intent, intent_version)        ↓(Optional) Signals → Mapping        ↓DecisionInput        ↓Validation        ↓Completeness Check        ↓Rule Evaluation        ↓Decision Outcome        ↓Event Logging → Hash → Replay → Verification⚙️ Step-by-Step Flow1. Input EntryExecution requires:intent + intent_versionInput enters in one of two modes:Mode A — Direct Decision InputClient → API → DecisionInputUsed when input is already structured.Mode B — Signal-Based InputRaw Signals → Ingestion → Mapping → DecisionInputUsed when input originates from external systems.2. Signal Processing (if applicable)Raw Signal Batch        ↓Ingestion (validation + normalization)        ↓Mapping (signals → schema fields)        ↓DecisionInput.system_data3. Decision InputFinal input passed to the core:{  user_input?: Record<string, any>  system_data?: Record<string, any>}Constraintsmust conform to schemaonly system_data is used in rule evaluationunmapped or unknown fields → INVALID4. ValidationPerformed using schema.Checks:field existencedata typesstructural correctnessOutcomeINVALID → stop executionNo rules are evaluated.5. Completeness CheckChecks:required schema fields are presentOutcomeINCOMPLETE → stop executionNo rules are evaluated.6. Rule EvaluationRules are evaluated in deterministic order:group → orderExecution stops at first matching rule.OutcomesRule matched → ALLOW / BLOCKNo rule matched → ESCALATE7. Decision Result{  status: "INVALID" | "INCOMPLETE" | "DECIDED",  decision?: "ALLOW" | "BLOCK" | "ESCALATE",  rule_id: string | null,  schema_version: string,  rule_version: string,  explanation: {...}}8. Debug Trace (Optional)If enabled, result may include:validation resultsmissing fieldsevaluated rulesmatched rulePurposedebuggingobservabilityaudit clarity9. Event LoggingEvery decision produces an immutable event:DecisionInput + DecisionResult + timestamp → hash → storedGuaranteesappend-only logdeterministic hashingno mutationaudit trail preserved10. Replay (Verification)Stored events can be replayed:Stored DecisionInput → re-execute → compare DecisionResultGuaranteesSame input → same outputFailure ConditionMismatch → SYSTEM INTEGRITY FAILUREIndicates:non-determinismschema/rule driftsystem violation🔐 System BoundariesOutside CoresignalsAPI layerintegrationsAI systemsInside Corevalidationcompleteness checkrule evaluationdecision outputCore ConstraintsCore MUST NOT:fetch external datacall APIsuse AImutate stateCore ONLY:DecisionInput → DecisionResult🧠 Core PrincipleDecision = Pure FunctionNo side effects.🔥 Full Flow Summary(intent, intent_version)        ↓(Optional) Signals → Mapping        ↓DecisionInput        ↓Validation → INVALID        ↓Completeness → INCOMPLETE        ↓Rules → ALLOW / BLOCK / ESCALATE        ↓DecisionResult        ↓Event Log        ↓Replay → Verify🚀 SummaryThe decision flow:enforces strict evaluation orderguarantees determinismseparates external data from core logicensures auditability and replayability
